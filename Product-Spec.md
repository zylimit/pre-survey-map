# Product Spec: Pre-Survey Map

> 站点勘测数据统一管理平台
> 版本：V1.0 · 日期：2026-05-28

---

## 产品说明

一个 B/S 架构的站点勘测数据统一管理平台，**唯一用户是勘测数据维护工程师**。

工程师每周把多源勘测数据（Google Earth 手工标记的 KMZ + 内部系统 ISDP / eSurvey 导出的 Excel）导入平台，做去重和冲突处理后入库；需要派工时，在地图上框选目标区域导出该区域 KMZ，通过邮件发给一线勘测员；勘测员用 Google Earth 打开 KMZ 直接看到任务位置去现场。

**核心价值**：统一散落在多个系统和手工标记里的勘测数据，**避免勘测员跑到已经勘测过的点上做重复工作**。

**用户**：
- 直接用户：**勘测数据维护工程师**（每周一次大批量维护 + 不定期派工，约 1-2 小时/次）
- 间接受益：**一线勘测员**（不直接使用平台，通过邮件接收 KMZ，用 Google Earth 查看）

## 功能需求

| ID | 功能 | 说明 |
|----|------|------|
| F1 | 单文件导入 | 支持 `.kml` / `.kmz` / `.xlsx`，点击按钮选择 + 拖拽到地图区；**V1 只允许一次一个文件**（拖入多个仅接受第一个并 warn）；单文件上限 100MB |
| F2 | 多源数据解析 | KML/KMZ 解析 Site / Road / Lessor 三类；Excel 解析 Site 详表（含 50 列扩展字段） |
| F3 | 冲突检测 | 按"大小写不敏感 + 去首尾空格 + 字符串比较"规则比对名称字段；同类比同类（点 vs 点、面 vs 面） |
| F4 | 冲突处理（步骤 2）| 数据清洗（F13）完成后进入；强制弹冲突列表对话框，不处理不让继续；单条/批量决策：覆盖 / 忽略（仅导无冲突） / 取消（导出冲突 Excel 自行修改） |
| F5 | 冲突列表导出 | 当用户选"取消"时，导出冲突明细为 Excel，含库内外字段双方对照 + 来源文件名 |
| F6 | 地图可视化 | OpenLayers 渲染；OSM 默认底图，可切高德 / Esri 卫星图 |
| F7 | 图层树管理 | 左侧单层树，三个固定文件夹：Site / Road / Lessor；节点带数量 + 复选框显隐 + 顶部搜索框；**列表虚拟化**（固定行高，DOM 只渲染可视区，支撑 13000+ 节点不卡顿）|
| F8 | 属性查看 | 点击地图要素或树节点 → 右侧面板平铺显示该要素**所有字段**（V1 不分 tabs）；可 ✖ 关闭，**关闭后无常驻入口**，再次点击要素或树节点自动重新打开 |
| F9 | 框选导出 KMZ | 工具栏 [框选模式 ▾] 选自由多边形 / 矩形 → 在地图绘制 → 导出严格包含在选区内的所有要素 |
| F10 | 整库导出 KMZ | 工具栏 [导出 KMZ ▾] 下拉：[导出整库] / [导出选区] |
| F11 | 操作反馈 | 底部可折叠输出面板：进度条 + 操作日志 + 错误警告 + 数据库连接状态 |
| F12 | 自动定位（双向焦点同步）| 导入完成自动 fit bounds 到新导入的数据范围；**点击树节点 → 地图飞到该要素并放大**；**地图选中要素 → 左树自动展开所在文件夹 + 滚动到该节点并高亮**（虚拟化列表下需按行号计算 scrollTop）|
| F13 | 数据清洗向导（步骤 1）| 解析后扫描脏数据 4 类：坐标写反 / 漏小数点 / 在海里 / 不在主基准；按类型分配默认动作；用户逐条或批量决策 [自动修复] / [保留] / [丢弃]；顶部 banner 显示**主基准区域 + 文件内自冲突去重摘要**；完成后进入步骤 2 冲突检测 |
| F14 | 清除基线（调试用 + 换基线唯一通道）| Toolbar 加 [🗑️ 清除基线] 按钮 → 弹确认 modal → 后端 truncate `site` / `road` / `lessor` + **`baseline_state`** 四表；**这是用户换基线国家的唯一方式**（V1 不做基线漂移、不做工作空间）；V1 不做权限控制 |
| F15 | 全局基线状态栏（V1.x #15）| Toolbar 下方独立一条 28px 横条，永远显示当前基线状态："📍 基线国家：菲律宾 (PH) · 固化于 2026-05-29" 或 "📍 基线未确立 · 首次导入完成后将自动锁定"|
| F16 | 全局搜索结果列表（V1.x #16）| Toolbar 右上搜索框回车 → **三类要素全搜（显示名匹配，与左树过滤同口径）**；结果写入底部 Output 的**独立"搜索结果区"**（不挤占 50 条日志），封顶 200 条；区头可点汇总条"🔍 搜索匹配 N 条，飞到第一条" + **[✖ 清空结果] 按钮**；每条结果一行 = **显示名 · 核心属性 · 类型角标**，可点 → **地图飞到该要素 + 左树该节点同步高亮滚动定位**（复用 F12 双向焦点同步）|

#### 全局基线状态栏（V1.x #15 · F15）

- **位置**：Toolbar 下方独立一条 28px 横条
- **背景**：浅灰底 (`#f8f9fa`)，accent 色边框 1px
- **文案模式**：
  - 已确立：「📍 基线国家：菲律宾 (PH) · 固化于 2026-05-29」（emerald 文字）
  - 未确立：「📍 基线未确立 · 首次导入完成后将自动锁定」（灰字）
- **数据来源**：单次 `SELECT * FROM baseline_state LIMIT 1`，启动 + 每次 commit 后刷新
- **点击行为**：V1 不做（V2 候选：点击显示基线详情 popup）

**宽度分配（默认）**：左树 20% / 中地图 55% / 右属性 25%。属性面板默认隐藏，选中要素后展开。底部输出面板默认折叠（28px），展开 200px。

**面板可拖拽缩放**：左树、右属性、底部输出三个面板的边缘有 **6px 拖拽条**（三个面板统一宽度，常驻边框色、hover/拖动时变 accent 绿），鼠标变成 `↔` / `↕` 光标时按住可调整大小；拖动时地图区自动 invalidate 重绘以匹配新尺寸。**拖动重绘经 `requestAnimationFrame` 节流**（避免 13000 要素下每像素一次 `updateSize` 导致拖动卡顿）。缩放范围：

| 面板 | 最小 | 最大 |
|------|------|------|
| 左树 | 200px | 500px |
| 右属性 | 240px | 600px |
| 底部输出（展开态） | 120px | 500px |
| 底部输出（折叠态） | 28px（固定） | — |

各面板缩放后的尺寸通过 `localStorage` 持久化，下次打开恢复。

**默认值策略（渐进式）**：
- **初次访问**（无 localStorage 值）→ 按 CSS 默认百分比渲染（左 20% / 右 25%），跨视口响应式
- **用户首次拖动后** → 该面板尺寸切换为 px 并写入 localStorage
- 之后该面板永远按 localStorage 的 px 值渲染，不受视口大小影响
- 清空 localStorage → 回到响应式百分比默认

| 按钮 | 操作 |
|------|------|

| 💾 导出 KMZ ▾ | 下拉：[导出整库] / [导出选区]（"导出选区"在未画选区时灰禁） |
| ⬛ 框选 ▾ | 下拉：[自由多边形] / [矩形]；进入绘制状态，鼠标变成十字 |
| 🔍 搜索 | 全局搜索框（右上角）；**回车触发**，三类要素全搜（Site/Road/Lessor，匹配各自显示名，与左树过滤同口径）→ 结果列入底部 Output「搜索结果区」+ **自动飞到第一条**（详见 F16 / 「底部输出面板·搜索结果区」）|

### 左侧树

- 顶部搜索框，实时过滤节点
- 三个**固定单层文件夹**：📂 Site / 📂 Road / 📂 Lessor，文件夹后括号显示数量
- 每个节点带复选框（默认勾选 = 显示在地图上）
- 文件夹复选框三态（全勾 / 半勾 / 全不勾），可批量控制整组
- 单击节点：地图飞到该要素（中心 + 自动放大） + 右侧属性面板显示
- **地图→树 反向定位**：在地图上选中要素时，左树自动滚动到对应节点并高亮选中；若该节点所在文件夹处于折叠态，先自动展开再滚入视口（与"单击节点→飞到地图"形成双向焦点同步）
- 面板宽度可拖拽（右边缘拖拽条，范围 200–500px，`localStorage` 持久化）
- 右键菜单：**V1 不做**

**性能 · 列表虚拟化（零依赖）**：节点树用固定行高（24px）虚拟化渲染，DOM 中只保留视口可见的 ~30 行 + 8 行 overscan，节点总数无论 13000 还是更多，DOM 节点数恒定。
- 动机：实测 13000 个节点全量渲染时，整个界面（面板拖拽、节点点击切换）卡顿 ~0.5s；定位确认元凶是树的 13000 个 DOM 节点（与地图无关——去勾选清空地图但树展开仍卡，折叠树立即丝滑）
- 实现：滚动容器内一个撑高的占位层（`height = 总行数 × 24px`），可见行用 `transform: translateY` 绝对定位
- 约束：选中态变化只重渲可见行，不重建整个行数组；地图→树定位时按目标行号计算 `scrollTop` 居中滚入

### 中间地图

- 引擎：**OpenLayers**
- 底图：默认 OpenStreetMap；右上角控件可切「高德地图」「Esri 卫星图」
- 地图控件：
  - 右上：缩放 +/- 按钮、定位（fit bounds 到全部数据）按钮、底图切换
  - 右下：比例尺 + 当前鼠标经纬度
- 要素样式（与原 KML 一致）：
  - Site：Positive 绿 pin / Negative 黄 pin / Unknown 红 pin
  - Lessor：Unfriendly 红面 / Normal 黄面 / Friendly 绿面
  - Road：棕色线
- 交互层级：**L2（只查不改） + 框选绘制**
  - 点击要素 → 属性面板显示
  - 框选模式下绘制临时多边形/矩形 → 完成后可导出

### 右侧属性面板

- 默认隐藏；选中要素后从右侧滑入
- 顶部：要素类型 + 名称（如 "Site: V2152" / "Lessor: Mall of Asia"）
- 内容：**所有字段平铺**列出
  - Site：50 列全展示
  - Lessor：4 列全展示
  - Road：1 列全展示
- 关闭按钮：右上角 ✖
- **关闭后无常驻入口**：再次打开靠点击地图要素或左树节点；点击新要素时面板自动展开 + 切换到新要素内容
- 面板宽度可拖拽（左边缘拖拽条，范围 240–600px，`localStorage` 持久化）

### 底部输出面板

- **折叠态**（默认）：高 28px，显示最新一条消息 + 右侧 DB 状态圆点
  - 搜索后折叠态显示汇总行：`🔍 搜索匹配 N 条，飞到第一条`（可点 → 展开面板并飞到第一条）
- **展开态**：高 200px，含：
  - 📊 进度条：导入/导出/解析时显示百分比
  - 🔍 **搜索结果区**（见下）：与操作日志**并列的独立区块**，不计入 50 条日志
  - 📜 操作日志：时间戳 + 消息（最近 50 条）
  - ⚠️ 错误警告：红色高亮显示
  - 🟢 DB 连接状态：左下角圆点（绿正常/红断开）
- 操作：点折叠条切换展开/收起；右上角 [清空日志]（**只清日志，不动搜索结果区**）
- 展开态高度可拖拽（上边缘拖拽条，范围 120–500px，`localStorage` 持久化）

#### 搜索结果区（F16）

- **触发**：右上角 🔍 搜索框回车
- **匹配规则**：三类要素全搜（Site / Road / Lessor），匹配各自**显示名**（site_id+option / road property / lessor name），大小写不敏感、子串包含——与左树过滤框同口径
- **结果呈现**（一行一条）：
  - 区头：左侧 `🔍 搜索匹配 N 条，飞到第一条`（可点 → 飞到第一条）；**右侧 [✖ 清空结果] 按钮**
  - 其下每条结果一行，格式 = **显示名 · 核心属性 …… 类型角标**；点击任意一条 → **地图飞到该要素 + 左树该节点同步高亮并滚动定位**（复用 F12 双向焦点同步）
  - **核心属性（填满宽输出框留白，一行平铺）**：
    - Site：`状态: <SITE STATUS> · 项目: <PROJECT> · (lat, lng)`
    - Lessor：`类别: <Lessor Category> · 关系: <Relationship>`
    - Road：`起点: (lat, lng)`（取 LineString 几何首坐标；无则省略）
    - 字段缺失则该项省略，不显示空 `状态: —`
- **清理手段**：区头 [✖ 清空结果] 按钮 → 清空搜索结果区、回到无结果态；**与 [清空日志] 并列、互不影响**（清日志不动结果，清结果不动日志）
- **封顶**：最多列 200 条；超出顶部提示"仅显示前 200 条，请细化关键词"
- **自动飞第一条**：回车后结果非空即自动飞到第一条（= 汇总条默认动作已执行）
- **自动展开**：每次新搜索自动展开 Output 面板，确保结果列表 / 边界文案可见
- **边界文案**：
  - 0 条匹配 → 搜索结果区显示红字"未匹配到任何要素"
  - 空关键词 → 不搜索（搜索框为空时回车无动作）
- **覆盖策略**：新搜索覆盖旧结果（搜索结果区只保留最近一次搜索）

## 用户使用流程

### 主流程 1：每周数据维护（1-2 小时）

1. 工程师打开网页 → 看到地图上已有数据
2. 拿到本周新数据：Google Earth 手工标的 KMZ + ISDP/eSurvey 导出的 Excel（多份，**逐个导入**）
3. 点 [📁 导入] → 选**第一个**文件（V1 一次只能一个）
4. 系统解析 → 底部进度条显示 "解析中..."
5. **进入两步导入向导**（单 modal，顶部 step 进度条：[1. 数据清洗] → [2. 冲突检测] → [完成]）

   **步骤 1：数据清洗（F13）**
   - 后端扫描脏数据 4 类（详见「数据清洗规则」节），返回清洗列表
   - 列：类型 / 名称 / 来源文件名 / 问题原因 / 当前坐标 / 修复后预览（如适用）/ 默认动作 / 操作
   - 默认动作按"先入为主"分配：写反 → [自动修复]、漏小数点 → [丢弃]、在海里 → [丢弃]、**不在主基准 → [丢弃]**（野蛮粗暴）
   - **极端保护**：基线已确立但本文件 0 个点位在基线国家境内 → 清洗向导顶部**红色警告 banner**："本文件 0 个点位在基线国家 PH 内，按当前决策导入将一无所获。要换基线请用 [🗑️ 清除基线]"
   - 每行单条按钮（写反 = 3 个选项 `[自动修复] [原样保留] [丢弃]`；其他 3 类 = 2 个选项 `[强制保留] [丢弃]`）
   - 行底色随决策变化：绿 = 自动修复 / 黄 = 保留 / 灰 = 丢弃
   - 顶部 banner（两层信息）：
     - **第 1 行 · 基准区域**：「基准区域：菲律宾（来自基线 14/14 个点，100%）」或「基准区域：菲律宾（来自本文件 14/17 点位，82% · 首次导入将固化为基线）」
     - **第 2 行 · 文件解析摘要**：「文件解析 234 个 site；**文件内重复自动去重 12 组**（保留后者，丢弃 11 个）；待清洗 223 个，检测异常 X 条」
   - 顶部批量按钮：[全部自动修复] / [全部保留] / [全部丢弃] / [按默认动作重置]
   - 底部 [取消导入] / [下一步：冲突检测 →]

   **步骤 2：冲突检测（F4）**
   - 经过清洗后还活着的点参与冲突检测（被丢弃的点不算冲突）
   - 列：类型 / 名称 / 库里现状 / 导入文件 / **来源文件名** / 操作

   - 顶部批量 [全部覆盖] / [全部忽略] / [反选]
   - 底部 [← 返回步骤 1] / [取消（导出冲突 Excel 自己改）] / [确认导入]

6. 工程师在两步里逐条或批量决策（**只修改 UI 状态，不触发入库**）→ 步骤 2 点 [确认导入]

## 补充说明

### 去重规则

| 类型 | 主键字段 | 备注 |
|------|----------|------|
| Site（点） | `SITE ID` + `OPTION` 组合 | 大小写不敏感、去首尾空格、按字符串比较 |
| Lessor（面） | `fid` | 同上 |
| Road（线） | — | **V1 不做去重**（后续考虑） |

### 合并策略

| 来源 | 策略 |
|------|------|
| 同一文件内重复 | **后者自动覆盖前者**，无需用户确认；**清洗向导顶部 banner 第 2 行显示去重摘要**让用户知道丢了什么 |
| 同一批次多文件内重复 | **后者自动覆盖前者**（与"同一文件内"相同规则；解析阶段 dict 折叠后再与库对照） |
| 文件 vs 库 | **弹冲突列表强制处理**：覆盖 / 忽略仅导无冲突 / 取消导出冲突 Excel 自行修改 |

字段合并方式：用户选"覆盖"时，新数据**完整替换**库中旧记录（V1 不做字段级合并）。

### 数据库设计

PostgreSQL 16 + PostGIS，**业务表 3 张 + 状态表 1 张 + 地理数据表 1 张**：

| 表 | 主键 | 核心强类型列 | 扩展字段 | 几何字段 |
|----|------|------------|---------|---------|
| `site` | `(site_id, option)` 联合 | PROJECT / SITE STATUS / SITE ID / OPTION / LATI / LONGI | `extras JSONB`（Excel 50 列扩展统一存此） | `geom POINT SRID 4326` |
| `road` | 自增 `id` | Property | — | `geom LINESTRING SRID 4326` |
| `lessor` | `fid` | Lessor Name / Lessor Category / Relationship | — | `geom POLYGON SRID 4326` |
| **`baseline_state`**（V1.x #15 新增）| `id INT CHECK (id = 1)` 单行约束 | `iso_a2` / `name_zh` / `coverage_pct` / `points_used` / `established_at` | — | — |
| `countries`（地理数据，docker init 时加载）| 来自 Natural Earth | `iso_a2` / `iso_a3` / `name` / `name_zh` | — | `geom MULTIPOLYGON SRID 4326`（GIST 索引）|

**字段集策略（两层）**：
- **强类型核心列**：KML 公共字段直接建列，承担去重主键、索引、空间查询
- **`extras JSONB` 扩展列**：Excel 的 50 列长尾字段（联系人、合同、技术规格等）统一塞入 JSONB，避免硬编码 50 列导致表结构脆弱；后续不同来源 Excel 列扩展，JSONB 直接容纳，无需迁移
- **解析时 `extras` 必须排除核心强类型列**（如 `SITE ID` / `OPTION` / `LATI` / `LONGI` 等），避免属性面板出现"同字段重复显示"
- Lessor 的 `extras` 排除集合**同时包含规范拼写和历史拼写错误**（`Lessor Category` 与 `Lessor Cagegory` 都要排除）
- 属性面板渲染顺序：先展示强类型列，再展开 `extras` JSONB 的 key-value 对

**批量空间查询性能边界**：
- 清洗判定（如 `classify_points`）的 PostGIS `ST_DWithin / ST_Distance` 等批量查询，**必须用 `unnest($1::text[], $2::float8[], $3::float8[])` 数组传参**
- 不能用 `VALUES ($1, $2, $3), ($4, $5, $6), ...` 这种逐参数 placeholder：PostgreSQL 单次查询参数上限约 65535，asyncpg 实际限制 32767，导入大批量点位（如 10000+ Excel 行）会触发 `too many parameters`

**主基准缓存原则**：
- 主基准国家 **一次性固化** 到 `baseline_state` 表，**不在 import 流程中动态计算**
- 禁止重复触发 `_country_dist_in_db` 对整张 `site` 表做空间查询（之前实测 13000 行 ≈ 30 秒，是 V1.x #15 之前的主要性能瓶颈）
- 详见「主基准区域算法」节的状态机

**镜像选型**：`postgis/postgis:16-3.4`（不是 `pgvector/pgvector:pg16` — 本项目要 PostGIS 空间扩展，不要向量扩展）

### KML / KMZ 处理

- 导入：KMZ 是 KML 的 ZIP 压缩包，后端先解压（找 `doc.kml` 或首个 `.kml`）再解析
- 导出：始终输出 KMZ；KML schema 与示例 `Integrated_Libraries_*.kml` 保持兼容（Site / Road / Lessor 三个 Schema）
- 导出 KMZ 内置 7 个 Style 定义（point-green / yellow / red、poly-green / yellow / red、line-brown），与示例文件一致，保证 Google Earth 渲染效果
- **解析时按 `<SchemaData schemaUrl="#site|#road|#lessor">` 区分类型；schema 缺失时按几何类型兜底**（Point→site / LineString→road / Polygon→lessor）
- **`extras JSONB` 展开规则**：导出 KMZ 时 `extras` 字段全部展开到 `<ExtendedData><SchemaData>`，不丢字段；`<Schema>` 节的字段定义取**所有同类要素 `extras` 字段集的并集**（union），保证不同来源的扩展列都被纳入
- **自反一致性契约**：本平台导出的 KMZ 重新导入回本平台时，必须 **100% 命中冲突**（同类型已去重要素全部识别为重复）。这是导入器和导出器之间的核心契约，保证数据"出去"和"回来"格式完全等价。任何破坏此契约的改动都视为回归 bug
- **导出文件命名规则**：`export_full_YYYYMMDD_HHMMSS.kmz`（整库） / `export_region_YYYYMMDD_HHMMSS.kmz`（选区）

### 字段名兼容（已知历史拼写错误）

| 原始数据中的字段名 | 规范字段名（数据库 / Spec） | 来源 |
|------------------|--------------------------|------|
| `Lessor Cagegory`（错拼） | `Lessor Category` / `lessor_category` | 示例 KML `Integrated_Libraries_*.kml` |

解析器导入时必须做这层映射，保证存库字段统一为规范拼写。导出 KMZ 时**也使用规范拼写**，逐步纠正历史数据。

### 解析期异常（解析失败直接拒绝，不进清洗向导）

| 异常 | 检测方式 | V1 处理 |
|------|---------|---------|
| Excel 列名不匹配（如 `Site_ID` vs `SITE ID`）| 必填字段缺失 | 解析失败，输出面板红色提示，列出检测到的候选列名 |
| KML schema 标签缺失或不规范 | `<SchemaData>` 没有 `schemaUrl` | 按几何类型兜底分类（Point→site / LineString→road / Polygon→lessor） |
| 文件超过 100MB | 前端 / 后端 / nginx 三端校验 | 前端直接拦下；后端 HTTP 413 |
| 网络断开 / 导入中途失败 | 数据库事务异常 | 导入事务回滚，库不脏 |

### 数据清洗规则（F13 · 进清洗向导让用户决策）

解析成功后，扫描 4 类脏数据，按下表分配默认动作并展示给用户：

| 异常 | 检测条件 | 默认动作 | 可选操作 |
|------|---------|---------|---------|
| **坐标写反**（LAT/LONG 颠倒）| LATI 绝对值 > 90 但 LONGI 在合法范围 / 或两者交换后都合法 | **[自动修复]** | [自动修复] / [原样保留] / [丢弃] |
| **坐标漏小数点**（无法自动修）| LATI 绝对值 > 90 且不属于写反场景 / LONGI 绝对值 > 180 | **[丢弃]** | [强制保留] / [丢弃] |
| **在海里** | 1km buffer 内不在任何国家陆地（详见「底层地理数据」节 SQL）| **[丢弃]** | [强制保留] / [丢弃] |
| **不在主基准区域** | 在某国陆地 1km buffer 内但不在基线国家陆地 1km buffer 内 | **[丢弃]**（野蛮粗暴 · 强约束"先入为主"，跨境点须用户主动 [强制保留]）| [强制保留] / [丢弃] |

**自动修复语义**：

- 坐标写反 → 入库时 swap LATI / LONGI 字段值（`extras` 里同步反过来）
- 其他类型 V1 不做自动修复

### 主基准区域算法（"先入为主" · **一次性固化，永不重算**）

主基准是 **一次性事件**：基线第一次确立时算一次，固化进 `baseline_state` 表，**之后所有导入都直接读，永不重新计算**。这是为了：

1. **避免基线漂移**（10 + 20 + 13000 个点连续导入，若每次重算会让基线被最后那批数据"夺权"）
2. **彻底解决性能问题**（之前每次导入都要扫整张 site 表，13000 行 → 30 秒）
3. **强约束"先入为主"语义**：核心数据先入库，杂数据再补

#### 状态机

```
未确立 (baseline_state 空表)
    │ 第一次 commit 成功 + site 表新增 > 0
    │ 算 _country_dist_in_db (基于刚入库的 site)
    │ 若返回非 None → INSERT INTO baseline_state
    ↓
已确立 (baseline_state 有 1 行)
    │ 后续 commit：什么也不做，baseline_state 不变
    │ F14 清除基线：TRUNCATE baseline_state + site + road + lessor
    ↓
未确立 (回到初始)
```

#### 三种使用场景

| 时机 | 读取方式 | 性能 |
|------|---------|------|
| 导入开始时检查基线 | `SELECT * FROM baseline_state LIMIT 1` | ~1ms |
| 清洗判定 `not_in_baseline` | 单行 SELECT 拿 `iso_a2` 然后传给 `classify_points` | ~1ms |
| 全局基线状态栏渲染 | 同上 | ~1ms |

#### 极端情况

- **首次入库点全在海里 / 全无国家**：`_country_dist_in_db` 返回 None → `baseline_state` 不写入 → 状态保持"未确立"，下次导入仍是 current_file 模式
- **已确立后导入文件全部不在基线国家**：清洗向导默认全部 [丢弃]；**顶部红色警告 banner**："本文件 0 个点位在基线国家 PH 内，按当前决策导入将一无所获。要换基线请用 [🗑️ 清除基线]"
- **已确立后用户主动 [强制保留] 不在基线国家的点**：这些点入库但 `cached_country_iso_a2` ≠ 基线 → 后续清洗仍以基线为准（不会因为这些"杂种点"动摇主基准）

#### 清洗向导 banner（仍保留 + 极端保护）

- **基线未确立**：「基准区域：菲律宾（来自本文件 14/17 个点，82% · 首次导入将固化为基线）」
- **基线已确立**：「基准区域：菲律宾（已固化 · 来自基线 14/14 个点，100%）」
- **基线已确立 + 本文件 0 点在基线国家**：上方红色警告 banner（雷 29 选项 B）

### 底层地理数据（单表统一判定 + 1km buffer 容错）

| 数据集 | 用途 | 加载方式 |
|--------|------|---------|
| **Natural Earth `ne_10m_admin_0_countries`** | 国家陆地多边形（**同时**判"在海里"+"不在主基准国家"）| docker init 时一次性加载到 PostGIS 表 `countries`，关键字段：`iso_a2` / `iso_a3` / `name` / `name_zh` / `geom` |

体积约 30MB（随仓库 `deploy/db/geo_data` 提供 SQL dump 或 GeoJSON），CC0 公共领域 + 离线，不依赖外部 API。

**判定改成单表统一逻辑（避免两数据集间矛盾 + 容错小岛）**：

```sql
-- "在海里"判定
NOT EXISTS (SELECT 1 FROM countries WHERE ST_DWithin(point.geom, countries.geom, 0.01))
-- 0.01 度 ≈ 1.1km 近赤道。1km buffer 容错 Natural Earth 10m 分辨率漏掉的小岛/礁石

-- "不在主基准"判定
EXISTS (SELECT 1 FROM countries WHERE ST_DWithin(point.geom, countries.geom, 0.01))
  AND NOT EXISTS (
    SELECT 1 FROM countries
    WHERE iso_a2 = :baseline_iso_a2 AND ST_DWithin(point.geom, geom, 0.01)
  )
-- 即：在某国陆地 1km 内，但不在基线国家陆地 1km 内
```

**为什么不用 `ne_10m_land`**：

- Natural Earth `ne_10m_admin_0_countries` 已经覆盖所有有主权的陆地（含小岛）
- 国家边界数据维护通常比纯陆地数据更精细（主权诉求导致岛屿覆盖更全）
- 单表判定避免"两数据集结果矛盾"（如：land 说在海里、countries 说在某国）
- 体积更小

## V2 候选功能

### 工作流类
- **基线 / 个人空间双工作区**（Git-like staging area）
  - 工程师所有导入/编辑操作只影响个人空间副本，与基线对比 diff，确认后批量同步（commit）到基线
  - 动机：① 避免频繁写后端库；② 提交前可视化对比变更
  - 设计待定：个人空间存储形态（前端 IndexedDB vs 后端 workspace 表）、身份识别方式、同步冲突解决、派工导出从哪个空间
- **多人协同 + 冲突解决**：操作日志、并发锁、回滚
- **认证 / 鉴权 / 账号体系**：与个人空间和多人协同配套
- **历史版本**：基线数据的时间快照
- **工作空间（一空间 = 一国家）**（V1.x #15 用户提出）
  - 当前 V1 是"一个数据库 = 一个国家"，换基线必须 F14 清除
  - V2 允许同时维护多个国家的勘测库：菲律宾空间 / 印尼空间 / 越南空间 ...
  - 顶部基线状态栏改为下拉切换空间
  - 每个空间独立的 `baseline_state` + 各自的 `site`/`road`/`lessor`（schema 隔离或 tenant_id 字段）
  - F14 清除按空间维度

### AI 能力类

### 视觉一致性约束（实施硬要求）

- 所有元素**禁止硬编码色值**，必须引用 CSS Variables
- CSS Variables 体系**只换"值"不换"键"**，组件代码理论上零修改
- **双主题实现（#19）**：暗色为默认值，定义在 `:root`（或 `[data-theme="dark"]`）；亮色定义在 `[data-theme="light"]`。两套同名令牌只是值不同。切换 = 改 `<html data-theme>` 属性，`localStorage` 持久化，**默认 dark**
- Toolbar 右侧加 **☀/☾ 切换按钮**（唯一新增控件；其余交互文案、布局、数据规则、API、命名规则、事务边界完全不动）
- 已有组件（Toolbar / LayerTree / MapView / AttributePanel / OutputPanel / ConflictDialog 等）逐项核对，确保无硬编码色漏网
- **地图区不参与主题（见「地图区护栏」）**：MapView / 底图 / 要素样式不动
- 字体加载方式不变
